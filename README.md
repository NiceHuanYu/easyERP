本项目面向约 20～200 人的小型制造企业。系统采用“订单驱动生产”的主线：销售订单贯穿计划、采购、生产、库存、发货和结算，实现业务数据统一、流程可追溯、部门协同。

## 1. 设计目标

- 以销售订单为业务主线，形成从接单到回款的闭环。
- 用统一的单据状态和审批流程控制业务动作。
- 库存变动必须产生流水，保证实时库存与可追溯性。
- 采用模块化、事件驱动的设计，便于后续扩展 MES、WMS、CRM 和 BI。
- 初期可单体部署，业务增长后可逐步拆分服务。

                         │
                   ERP Web / APP
                         │
 ┌──────────────────────────────────────────────────────────┐
 │                         业务服务层                         │

### 分层职责

| 层级 | 职责 |
| --- | --- |
| Web / APP | 面向各岗位提供录入、查询、审批和看板界面。 |
| API / 控制层 | 接收请求、身份认证、参数校验和返回结果。 |
| 应用与业务服务层 | 处理销售、采购、库存、生产、质量和财务等业务规则。 |
| 工作流引擎 | 控制审批、状态转换、退回、取消、超时提醒和通知。 |
| 事件总线 | 发布并订阅业务事件，降低模块之间的直接依赖。 |
| 数据访问与数据库 | 持久化单据、主数据、库存流水、审批日志和财务数据。 |

## 3. 功能模块

| 模块 | 核心职责 |
| --- | --- |
| 基础资料 | 管理物料、产品、BOM、客户、供应商、仓库、单位和价格等主数据。 |
| 销售 | 管理询价、报价、销售订单、发货、退货和订单进度。 |
| 采购 | 管理采购申请、审批、采购订单、到货、退货、对账和付款。 |
| 仓库 | 管理原材料、半成品、成品、退货和废料库存，以及出入库、调拨、盘点。 |
| BOM / MRP | 维护产品配方或物料清单；按需求、库存和交期计算净需求。 |
| 生产 | 管理生产计划、工单、排程、领退料、工序报工和完工入库。 |
| 品质 | 支持来料检验（IQC）、过程检验（IPQC）和出货检验（OQC）。 |
| 财务 | 管理应收、应付、发票、收付款、成本及订单/客户利润分析。 |
| 权限 | 基于角色控制菜单、数据范围和操作权限，并保留审计记录。 |
| 数据分析 | 提供订单交付、库存周转、采购交期、生产进度、质量和经营看板。 |

## 4. 订单驱动的端到端流程

```text
客户询价
    │
    ▼
销售报价
    │
    ▼
销售订单（SO）
    │
    ▼
订单审核
    │
    ▼
生产需求 / 主生产计划（MPS）
    │
    ▼
物料需求计划（MRP）
    │
    ├───────────────────┐
    ▼                   ▼
采购申请 → 采购订单 → 到货 / IQC → 原料入库
                                        │
生产工单 → 排程 → 领料 → 报工 → 质检 ───┘
    │
    ▼
成品入库
    │
    ▼
销售发货
    │
    ▼
开票、收款与订单关闭
```

### 核心处理规则

1. 已审核销售订单生成生产需求；系统先检查可用库存，再执行 MRP。
2. MRP 根据 BOM、现有库存、已分配库存、在途采购和安全库存计算净需求，生成采购申请和生产工单建议。
3. 采购到货须经 IQC；合格物料入库，不合格物料进入退货、返工或让步接收流程。
4. 生产工单领料、退料、报工、质检和完工入库均关联工单与批次。
5. 发货前检查订单状态、可发库存与出货检验；发货成功后生成应收依据。
6. 所有影响库存的业务均通过库存流水处理，禁止直接修改库存余额。

## 5. 单据状态机

以销售订单为例，使用状态机而非分散的布尔字段管理单据生命周期。

```text
草稿（Draft）
    │ 提交
    ▼
待审核（Pending）
    │ 审批通过
    ▼
已审核（Approved）
    │ 下达生产 / 备货
    ▼
生产中（In Production）
    │ 完工且可发货
    ▼
待发货（To Ship）
    │ 出库发运
    ▼
已发货（Shipped）
    │ 签收、结算完成
    ▼
已完成（Completed）
```

```text
异常分支：

草稿 / 待审核 ──取消──► 已取消（Cancelled）
待审核 ──退回修改──► 草稿（Draft）
已审核及后续状态 ──按权限关闭──► 已关闭（Closed）
```

采购申请、采购订单、生产工单、检验单、入库单、出库单、发票和付款单都应采用相同原则：明确状态、允许的转换、转换人、时间和审计日志。

## 6. 库存流程

```text
库存变化请求
      │
      ▼
校验单据状态、仓库、批次与可用库存
      │
      ├──不足 / 不合法──► 拒绝并返回原因
      │
      ▼
生成库存流水
      │
      ▼
更新库存余额与可用量
      │
      ▼
关联来源单据并写入审计日志
```

库存流水至少应记录：时间、物料、仓库/库位、批次、数量、单位、入出库类型、来源单据、去向单据和操作人。典型类型包括采购入库、生产领料、生产退料、完工入库、销售出库、调拨、盘点调整和报废。

## 7. 采购与生产流程

### 采购

```text
MRP / 人工需求
    ▼
采购申请
    ▼
采购审批
    ▼
采购订单
    ▼
供应商确认与到货
    ▼
IQC 来料检验
    ├──不合格──► 退货 / 返工 / 让步接收
    ▼ 合格
采购入库
    ▼
应付对账、发票与付款
```

### 生产

```text
销售订单 / 预测需求
    ▼
生产工单
    ▼
生产排程
    ▼
领料
    ▼
工序加工与报工
    ▼
过程检验（IPQC）
    ▼
完工检验
    ▼
成品入库
```

工单可以拆分为切割、冲压、焊接、喷涂、装配、测试、包装等工序；每道工序独立记录计划数量、完工数量、不良数量、工时、设备/人员和状态。

## 8. 工作流与事件驱动

工作流引擎负责审批、状态流转、通知、日志与超时提醒；业务模块负责领域规则与数据处理。

| 事件 | 触发时机 | 典型订阅动作 |
| --- | --- | --- |
| `OrderCreated` | 创建销售订单 | 通知销售、记录日志。 |
| `OrderApproved` | 销售订单审核通过 | 生成生产需求、触发库存检查和 MRP。 |
| `PurchaseArrived` | 采购到货登记 | 创建 IQC、生成待入库记录、更新应付依据。 |
| `WorkOrderFinished` | 工单完工 | 创建质检任务、更新订单进度、准备成品入库。 |
| `ShipmentFinished` | 销售发货完成 | 更新订单状态、生成开票/应收依据、通知客户。 |

事件处理应具备幂等性，并保留失败重试和审计记录，避免重复消息造成重复出入库或重复记账。

## 9. 接口处理链路

```text
用户操作
    ▼
Controller / API
    ▼
身份与权限校验
    ▼
Service（业务规则）
    ▼
Workflow（状态转换校验）
    ▼
Repository（数据访问）
    ▼
Database
    ▼
发布领域事件、写审计日志、发送通知
```

例如“发货”操作依次校验订单状态、可发库存与质检结果；在同一业务事务中生成出库单和库存流水、更新订单状态；事务成功后发布发货事件并生成应收依据。

## 10. RBAC 权限模型

```text
用户登录
    ▼
身份认证（Authentication）
    ▼
加载用户角色（Role）
    ▼
加载权限与数据范围（Permission）
    ▼
访问菜单、查询数据、执行操作
    ▼
记录关键操作审计日志
```

| 角色 | 主要权限 |
| --- | --- |
| 销售 | 维护客户、报价、销售订单、发货申请及订单查询。 |
| 采购 | 处理采购申请、采购订单、供应商与到货跟进。 |
| 仓库 | 执行收货、入库、领料、出库、调拨和盘点。 |
| 生产 | 查看工单、领取物料、报工和反馈异常。 |
| 质量 | 执行 IQC、IPQC、OQC，处理不合格品。 |
| 财务 | 管理发票、收款、付款、应收应付和成本报表。 |
| 管理员 | 管理组织、用户、角色、权限、流程配置与系统参数。 |

权限应同时覆盖功能权限（能否操作）、数据权限（可见哪些组织/仓库/单据）和审批权限（可审批哪些金额或单据类型）。

## 12. 推荐技术架构与实施路径

建议采用 **DDD（领域驱动设计）+ 工作流引擎 + 事件驱动架构**。领域边界可以划分为销售、采购、库存、生产、质量、财务和身份权限；模块通过应用服务、领域事件和 API 协作。

```text
前端（Web / APP）
       │
API Gateway / 统一认证
       │
┌────────────────────────────────────────────────────────┐
│ 应用服务：Sales │ Purchase │ Inventory │ Production │ QC │ Finance │
└────────────────────────────────────────────────────────┘
       │
工作流引擎  ───── 事件总线
       │              │
┌────────────────────────────────────────────────────────┐
│ 领域模型：Order │ WorkOrder │ Inventory │ Finance │ Master Data │
└────────────────────────────────────────────────────────┘
       │
MySQL / PostgreSQL + 对象存储（附件）+ 日志与监控
```

实施上建议分阶段推进：

1. **第一阶段：业务闭环** — 基础资料、销售订单、采购、库存、生产工单与出货。
2. **第二阶段：现场优化** — MRP、排程、条码、批次追溯和 IQC/IPQC/OQC。
3. **第三阶段：经营分析** — 财务集成、成本核算、利润分析、BI 和设备维护。

该方案能在保持小型制造企业系统简洁的同时，为未来规模化与系统集成保留扩展空间。
